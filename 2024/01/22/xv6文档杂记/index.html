使用fork系统调用后，父进程和子进程两个进程会并发地执行fork()语句后面的代码。当pid不为0时，该进程为父进程；当pid为0时，该进程为子进程。

OS执行用户程序的过程：

  1. 父进程调用fork生成子进程，父进程调用wait等待子进程调用完毕；
  2. 子进程执行相应的用户程序：runcmd->exec

那么可以这样理解：shell本身就是一个进程，而运行在shell之上的应用程序都属于shell的子进程。

exec需要传递两个参数，一个是命令的类型，一个是命令的参数。

exec takes two arguments: the name of the file containing the executable and an array of string arguments.

如果exec执行成功，那么就不会返回调用的程序。

When exec succeeds, it does not return to the calling program; instead, the instructions loaded from the file start executing at the entry point declared in the ELF header.

0x1 第一个进程

进程作为调度的基本单位，它拥有自己独立的地址空间，且本质上是存储在内存中的。进程一经创建，就会被分配内存资源，用于存储自身的程序指令和数据。

与内存类似，进程的内存空间也可被划分为内核区和用户区，用户区用于保存程序必要的指令和数据，内核区用于保存内存内核区中的一些系统调用的指令和数据。页表的本质作用是将逻辑地址映射为物理地址，因此进程可通过页表访问内存中实际的内存单元。

0x3 陷入，中断和驱动程序

xv6的系统调用流程：https://www.cnblogs.com/lilpig/p/17168211.html

  1. 在用户模式，调用C函数库中的系统调用的封装，如write

  2. 该封装中会将具体的系统调用号加载到a7中，然后调用ecall指令

  3. ecall指令会做三件事:

    1. 切换UMode到SMode
    2. 将用户pc保存到satp
    3. 跳转到stvec指定的位置，也就是trampoline

  4. 在trampoline中

    1. 将用户寄存器保存到进程的trapframe中
    2. 从trapframe中读取内核栈、内核页表、中断处理程序usertrap的位置，当前CPU核心id
    3. 加载内核栈到sp、切换satp位内核页表，跳转到陷阱处理程序

  5. 陷阱处理程序将trapframe中的epc写成ecall的下一条指令，调用syscall执行系统调用

  6. syscall调用具体的在内核中的系统调用代码，然后将返回值写到a0

  7. usertrap执行usertrapret，为返回用户空间做准备

    1. 比如设置进程trapframe中的内核相关的信息，内核栈、内核页表、中断处理程序位置等
    2. 设置sret指令的控制寄存器，以在sret执行时顺利恢复到用户模式
    3. 设置sepc为trapframe的epc
    4. 使用函数指针，调用trampoline代码中的userret，并将TRAPFREAM作为a0、用户页表位置作为a1

  8. trampoline中的userret做的就很简单了，切换回用户页表，从trapframe恢复用户寄存器

  9. 执行sret返回到UMode

大体上是这样的流程

自述整体流程

用户调用写在user.h中的C函数定义，代码跳转到usys.S中的汇编代码，这个汇编代码会将将相关的系统调用编号加载到a7寄存器中（li a7, SYS_write），然后执行ecall。

ecall指令会让代码跳转至trampoline中，该代码段用于实现User Mode和Supervisor Mode之间的转换。在这里，我们需要将用户调用时寄存器的值保存到tramframe中（sscratch寄存器指向这个结构体），这个结构体保存了进程的数据，包括通用寄存器的值、内核页表地址、内核栈地址、CPU核心ID等数据。

保存完寄存器的值后，我们从trapframe中提取内核环境（内核页表地址、内核栈地址等），标志着运行环境从User Mode成功转换为Supervisor Mode。

此时，在trampoline中执行jr t0指令，跳转至usertrap()函数的首地址，内核开始处理这个软中断。

代码进入trap.c，在这里：

  • 设置中断、异常处理程序的首地址：w_stvec((uint64)kernelvec);
  • 判断中断来源：if(r_scause() == 8){ ... }，确保中断来源是用户的系统调用请求。

当确认是来自用户系统调用请求时，系统会打开中断（intr_on()），并执行相应的系统调用（syscall()）。

代码进入syscall.c，在这里通过p->trampoline->a7获取到要执行的系统调用号，并通过它来索引一个存储了所有系统调用的函数指针数组，并跳转到对应系统调用函数的首地址进行函数调用。最后返回值存储在p->trampoline->a0中。

此时系统调用执行完毕，我们调用usertrapret()函数，将相关的内核环境重新保存至进程的tramframe中，并开始恢复用户环境（用户页表首地址、用户栈首地址等），然后回到trampoline中的userret函数中，进行所有通用寄存器值的恢复工作。

最后，执行sret函数，代码返回至User Mode，pc寄存器跳转到usys.S中的ret处，再次返回，整个系统调用到此结束。
